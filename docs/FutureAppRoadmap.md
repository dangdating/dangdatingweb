# 댕개팅(Dang-Dating) — Future App Roadmap

## 0. 목적
- 본 문서는 랜딩 이후 앱 본체 확장을 위한 단계별 로드맵입니다.
- 랜딩 구현 기준/우선순위는 [LandingPlan.md](/C:/Users/gmdqn/dang/docs/LandingPlan.md) 를 따릅니다.

## 1. 범위 원칙
- 본 문서는 앱 기능 확장 계획만 포함합니다.
- 랜딩 API(`/api/lead`, `/api/event`)와 앱 API는 분리 운영합니다.
- 랜딩에서 수집한 `Lead/Event`는 초기 유저 풀/퍼널 분석 목적으로만 사용합니다.

## 2. 단계별 로드맵
### Phase 1. 인증/프로필
- 계정/로그인
- 보호자/반려견 프로필
- 지역 인증 기본 체계
- 목표: 신뢰 가능한 기본 신원 레이어 확보

### Phase 2. 친구/채팅
- 보호자 연결 및 친구 요청
- 1:1 채팅
- 만남 제안/약속 보조 UX
- 목표: 동네 기반 연결 활성화

### Phase 3. 돌봄/후기/신뢰 점수
- 돌봄 요청/수락 플로우
- 후기/평가 시스템
- 신뢰 점수 및 뱃지 규칙
- 목표: 단발성 만남에서 반복 가능한 신뢰 구조로 전환

### Phase 4. 가족모드/커뮤니티
- 가족 모드 진입 기준
- 소규모 커뮤니티 그룹
- 오프라인 모임/이벤트 확장
- 목표: 장기 리텐션과 커뮤니티 네트워크 형성

## 3. 랜딩과 앱의 인터페이스 경계
- 랜딩은 전환 퍼널, 앱은 관계/활동 운영 제품으로 역할을 분리합니다.
- 랜딩 단계에서 앱 핵심 기능(채팅/돌봄 실행)을 선구현하지 않습니다.
- 랜딩에서 생성된 데이터는 앱 온보딩 우선순위와 가설 검증에만 사용합니다.

## 4. 착수 게이트 (Landing 선행조건)
앱 본체 개발 착수 전 최소 게이트:
1. CTA 클릭률 목표 도달 또는 개선 실험 계획 확정
2. 사전 등록 전환율 및 품질(중복/스팸 비율) 검증
3. 폼 제출 안정성(실패율 기준) 충족
4. 랜딩 성능/접근성/Lighthouse 기준 충족

## 5. 기술/운영 메모
- 인증, 권한, 보안, 데이터 보존 정책은 별도 앱 PRD에서 확정합니다.
- 마이그레이션은 additive-first 원칙(가능한 추가 컬럼 중심)으로 진행합니다.
- 앱 런칭 전 베타 운영 정책(신고/차단/모니터링)을 문서화합니다.
